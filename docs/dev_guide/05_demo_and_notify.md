# 05 — Demo Replay and Order-Opportunity Notifications

> Router: `docs/dev_guide/webui.md`. Read this doc for 演示模式 (demo replay), 下单机会通知 (order-opportunity notify), and the Feishu/PushPlus trigger wiring.

## 1. Architecture

- **Demo replay backend**: `pa_agent/webui/services/demo_runner.py::DemoRunner`/`build_demo_steps()` — re-expresses the desktop `pa_agent/demo/replayer.py::DemoReplayer`'s **scheduling algorithm** (character-level streaming pace, `_CHAR_MS`/`_STAGE_GAP_MS` timing, stage ordering) as a sequence of `/ws/analysis` JSON messages, rather than importing `DemoReplayer` directly. **`DemoReplayer` itself is not importable/reusable as-is**: it's a `QObject` subclass whose scheduling (`QTimer.singleShot` recursive callbacks) is fundamentally incompatible with asyncio's cooperative model, even though instantiating it outside a `QApplication` event loop wouldn't itself crash (Qt signal `emit()` without an event loop is a documented no-op, same as `SessionTokenLedger` in phase 5). This is the canonical example of "algorithm-only reusable" from `webui.md` §1.1 — reuse the shape of the logic, not the class. Content-extraction helpers (`_prompt_parts()`, and `content_from_response()`/`reasoning_from_response()` from `pa_agent.ai.response_extract`) **are** imported directly, not re-copied.
- **Demo mode lives inside `/ws/analysis`**, not a separate WebSocket: `submit` messages gain `mode: "demo"` + `demo_record_id`; `analysis.py`'s `_run_demo()` branch produces the same `record`/`trade_metrics` message shape as a real analysis (see `02_analysis_and_decision.md`), plus a `demo_finished` message when the replay completes.
- **Record picker**: `GET /api/demo/records` (`pa_agent/webui/api/demo.py`) reads `records/pending/`, filtered by `is_demo_playable()` — **only returns playable records**, unlike the desktop's manual-file-picker path (which lets you pick any file, then pops an error + auto-reselects if it's not playable). The web version collapses that two-step "pick, discover it's invalid, reselect" desktop flow into "the list never offers an invalid choice" — same net effect, simpler implementation, no "select-then-fail-then-retry" branch needed.
- **Frontend**: no dedicated `src/demo/` panel component — the record-picker dropdown + play/random-play buttons live **inline in `Toolbar.tsx`**, not a separate stacked block. This was a deliberate phase-6 layout choice: a disabled placeholder button (`<button disabled title="阶段五开放">演示模式</button>`) left in the toolbar by phase 5 signaled "this control belongs in the toolbar," and adding a whole new panel row risked repeating the `.app-shell` layout-budget problem (see `08_state_and_layout.md`). Only pure formatting logic (`formatDemoRecordLabel`) lives in a standalone, unit-tested file, `src/demo/demoFormat.ts`.
- **"Random" replay** is plain frontend `Math.random()` over the already-fetched record list — not the desktop's `pick_playable_demo_record()` (which excludes the previous pick and prefers records with richer decision-tree data). Fine at small record counts; if `records/pending/` grows large enough that the dropdown becomes unwieldy, a server-side random-pick endpoint with smarter exclusion logic would need to be proposed separately.

## 2. Order-Opportunity Detection + Notification

- **`pa_agent/webui/services/order_alert.py::maybe_alert_order_opportunity()`** — ports the desktop `pa_agent/gui/order_opportunity.py`'s detect+broadcast half (imports `has_order_opportunity`/`format_order_alert_message` directly, pure functions, no Qt). Called from `analysis.py::_run()`/`_run_demo()` right after a record is produced — **runs unconditionally regardless of whether the record came from a real analysis or a demo replay**, matching the desktop's own `_on_analysis_finished` behavior (which doesn't special-case `self._demo_mode` either). Do not add a "skip for demo mode" branch without checking this is still what the user wants.
- Uses `decision_shape.decision_inner()` (see `02_analysis_and_decision.md`) to unwrap the nested-vs-flat shape before checking for an opportunity.
- Notification dispatch: `_spawn_notify()` (background thread) calls the **existing** `pa_agent/notify/feishu_notifier.py`/`pushplus_notifier.py` functions — zero changes to their signatures/retry/request format. The web layer only adds the *trigger*, not new notification logic.
- **Frontend**: `src/notify/OrderOpportunityToast.tsx` — page-in toast only. **No browser Notification API, no sound** (confirmed with the user in phase 6 — a deliberate scope limit, not a missing feature). `position: fixed` overlay, doesn't consume `.app-shell` vertical layout budget.
- Test coverage never exercises a real network call — `send_order_signal`/`pushplus_is_active` are mocked in both pytest and e2e. **No automated verification against a real Feishu webhook or PushPlus token exists** (a standing, explicitly-accepted gap — the user was asked in phase 7 whether this needed real-credential manual verification for final acceptance and said no).

## 3. Two Explicitly-Confirmed Non-Wirings — Do Not Wire Without Asking First

- **`notify_on_order_only`** (a `config/settings.json` field, exists since phase 1's settings schema): **never read/consumed by either the desktop GUI or the web layer.** Phase 6 explicitly chose not to introduce it (the desktop's own `_spawn_post_order_followup` never reads it either — wiring it up on the web side only would create *new* two-sided inconsistency, worse than the current "consistently unused" state). Phase 7 re-confirmed: keep as-is, documented, not deleted. **If you're asked to make this field do something, that's a decision for the user, not an inference to make yourself.**
- **`pa_agent/records/trade_logger.py::save_trade_record()` is not called anywhere in `pa_agent/webui/`.** The desktop's `_spawn_post_order_followup` writes a CSV row (feeding the `06_reports_dashboard.md` report page) *in addition to* sending the notification, whenever an order opportunity fires. The web layer currently only does the notification half. This was explicitly evaluated and declined in phase 7 (the desktop path also attaches a chart screenshot the web layer has no equivalent capability for — `chart_image_path` would need to be `None`, a real behavior gap, not just an implementation gap). **Confirm with the user before wiring this up** — it's flagged, not silently missing.
